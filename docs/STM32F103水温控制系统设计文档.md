# STM32F103 水温控制系统软件设计文档（V2.5）

## 2. 关键指标

- 控温范围：20℃~60℃（默认）
- 报警阈值：60℃（可配置）
- 采样周期：1 s
- 控制周期：1 s
- 显示刷新：200 ms
- 控温稳态误差：≤ ±0.5℃（合理保温结构 + PID 调参后）
- 数据记录周期：1 s / 5 s / 10 s（可选）

## 6. 软件设计

### 6.1 软件架构

- 软件栈固定为：`STM32F10x StdPeriph + 裸机前后台架构`，禁止引入 RTOS
- 设计原则：模块解耦、定时调度、状态机驱动、参数集中管理、异常优先保护
- 分层结构：
  - BSP/驱动层：GPIO / TIM / I2C / SPI / UART / 并口 TFTLCD / OneWire / RTC / SPI Flash
  - 设备抽象层：ds18b20 / display / relay / buzzer / key / rtc / spiflash / storage / uart_export
  - 服务层：soft_timer / key_service / ui_service / log_service / alarm_service / schedule_service / param_service
  - 控制层：temp_acquire / temp_filter / pid_ctrl / heater_ctrl / safety_ctrl
  - 应用层：main_app / menu_app / export_app

### 6.3 任务模型

- 本项目必须不用 RTOS，原因是任务规模有限、实时性要求明确、调试更简单
- 使用 `SysTick + 软件定时器 + 主循环状态机` 即可满足需求
- 禁止复杂调度策略（如多级优先级抢占、动态任务管理、运行时任务拓扑变更）
- 任务调度采用“快中断、慢任务”原则：中断只置位标志，不做复杂业务处理，主循环按固定顺序轮询服务模块

### 6.4 周期任务（建议）

- 1 ms：系统心跳、软定时器
- 100 ms：按键扫描、蜂鸣器状态机
- 200 ms：显示刷新
- 1 s：温度采样、PID 计算、继电器控制、日志记录、预约判断

建议主循环形式如下：

```c
while (1)
{
    scheduler_poll();
    key_service_process();
    ui_service_process();
    export_service_process();
    alarm_service_process();
}
```

### 6.5 模块职责划分

- `temp_manager`：读取 3 路 DS18B20，完成 CRC 检查、异常判定、滤波和控制温度计算
- `pid_ctrl`：根据目标温度和控制温度输出控制量，支持积分限幅与输出限幅
- `heater_ctrl`：将 PID 输出映射为时间比例控制窗口，驱动继电器并限制最小动作周期
- `alarm_service`：管理超温、探头故障、RTC 异常等告警，具备最高优先级
- `schedule_service`：根据 RTC 时间判断预约开始/结束，切换系统工作模式
- `param_store`：负责参数加载、CRC 校验、写入与默认值恢复
- `log_service`：定时记录温度、状态和报警事件，支持串口导出
- `ui_service`：负责显示页面刷新、菜单切换、参数编辑与提示显示
- `protocol_export`：实现与电脑的数据交互协议，支持日志导出和参数查询

### 6.5.1 设备驱动实现约束（BSP）

- 驱动层统一采用裸机轮询方式，不使用 RTOS 与中间件任务框架。
- 驱动实现保持简洁高效，避免动态内存与复杂调度逻辑。
- `bsp_ds18b20` 采用标准 OneWire 时序与 CRC 校验，支持独立 3 路采样与失败重试。
- `bsp_rtc` 采用标准备份域初始化流程，优先 LSE，失败回退 LSI，并使用备份寄存器标记初始化状态。
- `bsp_uart` 采用轮询收发，发送路径需具备超时保护，避免异常硬件状态导致主循环阻塞。
- `bsp_relay` 仅负责电平驱动与状态缓存，控制策略由上层 `heater_ctrl` 统一管理。

### 6.6 系统状态机

建议将系统抽象为以下运行状态：

- `IDLE`：待机，未加热，等待手动启动或预约到点
- `HEATING`：闭环控温运行中
- `SCHEDULED`：预约已启用，等待到达开始时间
- `ALARM`：超温或关键故障，强制停热并告警
- `EXPORT`：串口导出日志或调试数据

状态切换原则：

- `ALARM` 优先级最高，可从任意状态切入
- `SCHEDULED` 到点后进入 `HEATING`
- 预约结束、手动停止或故障恢复后返回 `IDLE`

### 6.7 温度处理

- 合法区间校验：-55℃~125℃
- 异常值剔除：CRC 错误/读数跳变过大
- 滤波建议：滑动平均（窗口 3~5）或中值滤波
- 控制温度建议：
  - 正常控制：`T_ctrl = (T1 + T2 + T3) / 3`
  - 安全判断：任一路超过阈值即触发报警，不仅依赖平均值
- 传感器故障策略：单路失效时提示降级运行；多路失效时禁止加热

### 6.8 PID 控温

离散位置式 PID：

$$
 u(k)=K_p e(k)+K_i\sum_{i=0}^{k} e(i)\Delta t+K_d\frac{e(k)-e(k-1)}{\Delta t}
$$

其中：

$$
 e(k)=T_{set}-T_{ctrl}
$$

- 输出限幅：$u(k)\in[0,100]$%
- 建议控制周期：`1 s`
- 说明：对水温这类慢热系统，`1 s` 控制周期通常足够；同时 DS18B20 在高分辨率下转换时间接近 `750 ms`，因此不建议在初版实现中盲目缩短控制周期
- 控制方式：时间比例控制（例如 10 s 窗口）
- 继电器保护：设置最小吸合/释放时间，减少抖动与寿命损耗
- 推荐控制窗口：`10 s`
- 推荐初始参数：`Kp=8.0, Ki=0.3, Kd=15.0`，后续依据水体容量与加热功率现场整定
- 必要保护：积分分离、积分限幅、输出限幅、防止小误差下继电器频繁抖动

PID 现场调试建议：

- 第一步仅开启比例项：先令 `Ki=0, Kd=0`，逐步增大 `Kp`，观察升温速度与振荡趋势
- 第二步加入积分项：在 `Kp` 基本稳定后缓慢增大 `Ki`，用于消除稳态误差
- 第三步少量加入微分项：若过冲明显，再增加少量 `Kd` 抑制温度变化过快导致的超调
- 建议每次只修改一个参数，单次调整幅度不超过 `10%`
- 每轮调试建议记录：超调量、整定时间、稳态误差、继电器动作频率

### 6.9 超温保护

- 当任一路温度 > 60℃：
  - 立即关闭加热
  - 蜂鸣器报警
  - OLED 显示超温提示
  - 记录报警事件
- 当 DS18B20 连续多次读取失败时，也应进入故障保护或降级运行模式

- 报警阈值
- 预约参数
- 记录周期

### 6.11 预约加热

- 可配置开始时间、结束时间
- 到点自动进入控温；到结束时间自动停止加热
- 页面显示预约状态（开/关、下一次执行时间）
- RTC 异常时禁止预约启动，并提示校时

### 6.12 数据记录与导出

- 本地环形日志记录：时间戳 + 三路温度 + 目标温度 + 输出 + 状态
- 串口支持两类用途：
  - 历史日志导出：用于调试后回放与保存 CSV
  - 实时数据上报：用于电脑端绘制温度/PID/继电器状态曲线
- 导出 CSV 字段：
  - timestamp,t1,t2,t3,t_avg,set_temp,pid_out,heater,alarm

实时绘图数据建议字段：

- `seq`：递增帧序号，用于检测漏帧
- `time_ms`：设备本地毫秒计数，用于恢复真实时间轴
- `t1,t2,t3,t_ctrl,set_temp,pid_out,heater,alarm,mode`

推荐实时数据帧示例：

```text
DATA:105,105000,24.81,24.76,24.79,24.79,45.00,62.50,1,0,1
```

其中字段顺序为：

```text
DATA:seq,time_ms,t1,t2,t3,t_ctrl,set_temp,pid_out,heater,alarm,mode
```

### 6.13 通信协议建议

- 物理层：UART，波特率 `115200`
- 协议层建议：采用基于 UART 的 SCPI 风格 ASCII 文本协议，便于串口助手直接调试和电脑端快速接入绘图脚本
- 工作模式建议同时支持：
  - 主机轮询模式：上位机定时发送查询命令，设备返回一帧数据
  - 流式上报模式：上位机发送 `STREAM:ON` 后，设备按控制周期主动输出 `DATA:` 帧
- 推荐命令：
  - `MEAS:TEMP?`
  - `MEAS:ALL?`
  - `READ:PID?`
  - `READ:STATE?`
  - `CONF:SETP 45.0`
  - `CONF:PID 8.0,0.3,15.0`
  - `LOG:EXPORT?`
  - `LOG:CLEAR`
  - `STREAM:ON`
  - `STREAM:OFF`
- 当前固件已支持的关键命令：
  - `READ_TEMP`
  - `READ_PARAM`
  - `SET_TEMP=<float>`
  - `SET_PID=<kp>,<ki>,<kd>`
  - `CONF:PID <kp>,<ki>,<kd>`
  - `SET_ALARM=<float>`
  - `CONF:ALARM <float>`
  - `SET_SCHEDULE=<en>,<start_min>,<end_min>`
  - `CONF:SCH <en>,<start_min>,<end_min>`
  - `SET_LOG_PERIOD=<sec>`
  - `CONF:LOGPERIOD <sec>`
  - `LOG_CLEAR`
  - `LOG_EXPORT`
- PID 下发约束：
  - `kp` 范围：`0.0~30.0`
  - `ki` 范围：`0.0~5.0`
  - `kd` 范围：`0.0~50.0`
  - 成功响应：`OK,PID_APPLIED`
  - 越界响应：`ERR,PID_OUT_OF_RANGE`
- 报警阈值下发约束：
  - 范围：`40.0~90.0`
  - 成功响应：`OK,ALARM_APPLIED`
  - 越界响应：`ERR,ALARM_OUT_OF_RANGE`
- 预约参数下发约束：
  - `en`：`0/1`
  - `start_min/end_min`：`0~1439`
  - 成功响应：`OK,SCHEDULE_APPLIED`
  - 越界响应：`ERR,SCHEDULE_OUT_OF_RANGE`
- 日志周期下发约束：
  - 范围：`1~60` 秒
  - 成功响应：`OK,LOG_PERIOD_APPLIED`
  - 越界响应：`ERR,LOG_PERIOD_OUT_OF_RANGE`
- 响应格式建议：
  - 查询命令返回单行 ASCII 数据
  - 设置命令统一返回 `OK` 或 `ERR,...`
  - 流式模式下仅输出 `DATA:` 开头的数据帧，避免与调试日志混发造成解析复杂化
- 轮询绘图示例：

```text
主机发: MEAS:ALL?
设备回: 25.31,25.44,25.28,25.34,45.00,36.50,1,0,1
```

- 流式绘图示例：

```text
主机发: STREAM:ON
设备回: OK
设备持续发: DATA:seq,time_ms,t1,t2,t3,t_ctrl,set_temp,pid_out,heater,alarm,mode
```

### 6.14 软件栈结论

本项目的软件栈建议固定为：`STM32F103rbt6 + STM32F10x StdPeriph + 裸机定时调度 + 分层模块化设计 + 状态机控制`。该方案的优点是实现直接、调试成本低、足够支撑 3 路测温、PID 控温、预约、报警、掉电记忆和数据导出，适合作为课程设计、毕业设计和实际打样的第一版实现。

## 7. 人机界面设计

### 7.1 主界面

- 时间（RTC）
- T1/T2/T3
- 平均温度
- 目标温度
- 加热状态
- 预约状态
- 报警状态

### 7.2 菜单结构

1. 目标温度设置
2. PID 参数设置
3. 报警阈值设置
4. 预约加热设置
5. 数据导出
6. 系统信息/恢复默认

### 7.3 交互逻辑（参考业界嵌入式实现）

- 交互范式采用“页面状态机 + 按键事件驱动 + 参数即时预览”的轻量模式。
- 事件模型：`SET/UP/DOWN/BACK` 短按、`SET` 长按、`UP/DOWN` 连发。
- 运行原则：
  - 非编辑态：`UP/DOWN` 切页，`SET` 进入编辑，`BACK` 回主页。
  - 编辑态：`UP/DOWN` 调整参数，`SET` 确认（PID 页用于字段切换），`BACK` 退出编辑。
  - 长按 `SET`：
    - 编辑态下快速退出编辑并返回非编辑态；
    - `INFO` 页面非编辑态下先进入恢复默认确认态，再次长按后执行恢复默认参数。
- 页面定义：
  - `HOME`：运行总览与状态提示。
  - `SET_TEMP`：目标温度调整。
  - `PID`：`Kp/Ki/Kd` 字段轮转编辑。
  - `ALARM`：超温阈值调整。
  - `SCHEDULE/EXPORT/INFO`：预约、导出与信息页面。

### 7.4 参数编辑约束

- 目标温度：`20.0~60.0℃`，步进 `0.5℃`。
- 报警阈值：`40.0~90.0℃`，步进 `0.5℃`。
- `Kp`：`0.0~30.0`，步进 `0.2`。
- `Ki`：`0.0~5.0`，步进 `0.05`。
- `Kd`：`0.0~50.0`，步进 `0.5`。
- 所有参数编辑必须执行边界钳位，防止越界输入导致控制异常。
- 编辑参数变更后需走统一 NV 延迟写入策略，避免高频擦写 Flash。

### 7.5 UI 驱动接入实现

- 按键接入：`ui_service_tick_100ms` 周期扫描按键状态，完成短按/长按/连发事件转换。
- 事件处理：使用小型环形队列串行处理按键事件，避免同周期多键抖动导致状态机错乱。
- 显示渲染：`ui_service_tick_200ms` 生成页面文本并调用 `bsp_oled_draw_text` 与 `bsp_oled_refresh` 刷新。当前计划保留这组文本 API，不改上层 UI 状态机，只替换底层显示实现。
- 页面渲染分层：
  - 第 1 行：模式与页面标识
  - 第 2 行：控制温度与目标温度
  - 第 3 行：PID 输出、加热与报警状态
  - 第 4 行：当前页面的参数摘要
- `EXPORT` 页面实现约束：
  - 显示当前日志条数与日志周期（秒）
  - 显示导出命令提示（`LOG_EXPORT`）
  - 不在 UI 线程直接触发导出，仅提供状态与操作提示
- `INFO` 页面实现约束：
  - 长按 `SET` 进入恢复默认确认态，再次长按才执行恢复默认参数操作（目标温度、报警阈值、PID、预约参数、记录周期）
  - 恢复后显示短时反馈提示，实际持久化由统一 NV 延迟写入路径完成
- 性能与交互平滑性约束：
  - 控制参数同步时，仅在 `Kp/Ki/Kd` 变更时重建 PID 上下文，避免因目标温度等参数调整造成控制突变。
  - 显示页面刷新采用变更触发策略，避免固定周期全量重绘造成闪烁与总线开销。
  - 按键事件队列在满载时采用“丢弃最旧事件”策略，优先保留最新交互输入，减少连按场景的丢键感知。
- 约束：渲染逻辑仅显示状态，不直接改变控制策略参数，参数修改只通过按键事件路径完成。
